# Project Context: Intelligent MCP Orchestration System

## Overview

An MCP (Model Context Protocol) server that accepts user input, builds a dynamic DAG (Directed Acyclic Graph) of tool tasks, executes them in parallel, and surfaces everything to the user through an advanced real-time GUI. The system features self-healing error recovery via auto-generated Sub-DAGs.

**My Role:** Frontend UI/UX — all frontend design, implementation, and user experience.

---

## Core Architecture

### 1. Input Layer
- User provides input via **plain text** or **image upload**
- If image: OCR extracts text from image → extracted text is passed to LLM
- LLM: **Kimi K2.5 1T** (primary model)

### 2. DAG Generation
- LLM receives the input and generates a **DAG (Directed Acyclic Graph)** of tasks
- DAG is visualized immediately in the GUI before/during execution

### 3. Tool Execution (Parallel)
All 4 tools run in parallel within the DAG:

| Tool | Purpose |
|------|---------|
| **Jira** | Task/issue management |
| **Slack** | Messaging / notifications |
| **GitHub** | Code / repo operations |
| **Google Sheets** | Audit logging of all actions |

### 4. Real-Time GUI
- Full live visualization of DAG execution
- Node states: pending → running → success / failed
- Every tool's progress shown in real-time
- Extremely advanced, polished UI/UX

## Error Handling & Self-Healing Sub-DAG System

When a task node fails, the system pauses the main DAG and classifies the error:

### Scenario 1 — Minor / Transient Error
- LLM retries the failed task automatically
- Maximum **3 retry attempts**
- If resolved within 3 tries → main DAG resumes
- Example: network timeout, temporary API hiccup

### Scenario 2 — Major but Resolvable Error
- LLM generates a **Sub-DAG** specifically to fix the error
- Sub-DAG runs independently to resolve the issue
- Already-completed tasks in the main DAG are **NOT re-executed** (no duplication)
- Once Sub-DAG completes → main DAG resumes from the failed node
- Maximum **3 attempts** for Sub-DAG resolution

### Scenario 3 — Sensitive / Unrecoverable Error
- System **immediately surfaces the error to the user** — no LLM fix attempt
- Examples: missing/invalid API key, auth failures, permission errors, quota exceeded
- User must manually resolve before the DAG can continue

> **Key Rule:** Sub-DAG generation is fully automatic and LLM-powered. Max 3 total resolution attempts across all retry scenarios.

## Key Constraints & Rules

- **Max retry attempts:** 3 (applies to all scenarios)
- **No task duplication:** Completed nodes are never re-run, even after Sub-DAG fix
- **Sub-DAG is automatic:** No manual user trigger needed
- **Sensitive errors bypass LLM:** Shown directly to user, no auto-fix attempted
- **Parallel execution:** All 4 tools execute concurrently within a DAG level

## Terminology Reference

| Term | Definition |
|------|-----------|
| **DAG** | Directed Acyclic Graph — the task execution plan generated by the LLM |
| **Sub-DAG** | A secondary DAG auto-generated by the LLM to fix a failed node |
| **Node** | A single task unit within a DAG (e.g., "Create Jira ticket") |
| **MCP** | Model Context Protocol — the server/orchestration layer |
| **Audit Log** | Real-time record of all tool actions, written to Google Sheets |
